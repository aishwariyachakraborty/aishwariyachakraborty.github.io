NVIDIA Profiling Tools

Published in February 2026

The increasing popularity of Nvidia GPUs can be partially attributed to the availability of a stable ecosystem of tools which aid in improving the usability of these accelerators. One such very important set of tools are the profiling tools, which provide fine grained visibility into the hardware as well as software stack built on top of it. This, in turn, helps to find out bottlenecks and pin-point causes of inefficiences in the performance of workloads running on these devices. In this blog, I describe a few of these existing profiling tools along with the functionalities they provide. Primarily, Nvidia provides three main profiling tools - Nsight Systems, Nsight Compute, and CUPTI. In addition, it also provides a system monitoring tool named Nvidia SMI, which is very useful for a systems engineer/researcher. Apart from these, there are a few other tools as well, such as Nvidia Data Center GPU Manager (DCGM) and Nsight Graphics, which I will (probably?) cover in a future blog.


Nsight Systems:
System wide performance analysis tool - helps to visualize workload and its resource usage in terms of various system metrics on a timeline, thereby aiding in identifying areas of performance improvement and tuning for scaling across any number of resources. It can track CPU and GPU events and correlate them to establish a timeline. It can look inside CUDA, cuBLAS, cuDNN, and NVIDIA TensorRT. Different metrics made available by Nsight Systems include PCIe throughput, NVIDIA NVLink®, and dynamic random-access memory (DRAM) activity, SM utilization, Tensor Core activity, instruction throughput, warp occupancy, and network metrics in multi-node cases. Nsight Systems supports two methods of code annotations to limit profile duration:

  • cudaProfilerStart() and cudaProfilerStop() APIs in the cuda_profiler_api.h header file to limit profiling to a region of a CUDA application.
  • use of NVIDIA Tools Extension (NVTX), an application interface to the NVIDIA Profiling tools, to annotate the profiler timeline with events and ranges and to customize their appearance and assign names to resources such as CPU threads and devices.

It can be used as a CLI, GUI, or as a Jupyter extension. After profiling, it generates a report file with .nsys-rep extension which can be analyzed further to get more information. We can also directly query NVML using c API to get the intermediate memory/compute usage values. For more info, go through the official documentation.


Nsight Compute:
An interactive kernel profiler for CUDA applications that collects detailed GPU performance metrics (via hardware monitors, kernel instrumentation, and replay) to analyze and optimize kernel performance. Frontend has both UI and CLI versions, which runs on the host. Host and target can be the same machine or different -- the remote target might as well have a different OS. Nsight Compute inserts its measurement libraries into the application process, which allow the profiler to intercept communication with the CUDA user-mode driver. The requested performance metrics are collected when a kernel launch is detected and the results are transferred to the frontend. All metrics may not be possible to collect in a single pass. In such cases, kernels or workloads may be replayed via several modes:

  • Kernel replay: All accessible memory for the kernel are saved and restored for each pass.
  • Application replay: All metrics requested for a specific kernel launch are grouped into one or more passes. The complete application is run multiple times so that in each run, metric values from one of those passes can be collected per kernel. Does not have memory save-and-restore overhead.
  • Range replay: Complete ranges of CUDA API calls and kernel launches within the profiled application are captured and replayed. The range is defined using Profiler Start/Stop API or NVTX Ranges.
  • Application range replay: A range of workloads is replayed by re-running the entire application without modifying interactions or saving and restoring memory. One pass is collected for each range in every application execution.

Nsight compute has a memory chart feature which shows a graphical, logical representation of performance data for memory subunits on and off the GPU. This includes data transfer sizes, hit rates, number of instructions or requests, etc. Many different metrics, including warp program counter and warp scheduler state, are collected using Nsight compute by sampling GPU's performance monitors periodically. Depending on the selected metric, data is collected either through a hardware performance monitor on the GPU, through software patching of the kernel instructions or via a launch or device attribute. Metrics which are hardware-provided and don’t have string “sass” in their names are collected with less overheads. On the other hand, those instrumented by software and having "sass" in their names are expensive to collect as CUPTI needs to instrument the kernel to collect these and these need to be collected in individual passes to prevent impact of instrumented code on other metrics. For more info, go through the official documentation.


CUPTI:
The CUDA Profiling Tools Interface or CUPTI ensures seamless profiling compatibility for CUDA applications across various GPU architectures and CUDA driver versions. It has both C and Python APIs, and can profile code both on CPU and GPU. CUPTI supports tracing, i.e., collection of timestamps and metadata related to various GPU/CUDA events, and profiling, i.e., collection of GPU performance metrics per kernel or set of kernels. PyTorch Profiler interacts with CUPTI under the hood to obtain the various events for trace generation. The various functionalities supported by CUPTI are as follows:

  • Enables low-overhead profiling by providing callback mechanism to notify subscribers of an event of its occurence.
  • Provides the option to profile a specific range within an execution.
  • Supports the sampling the warp program counter and scheduler state to identify reasons for stall.
  • Collects hardware metrics by sampling GPU performance monitors and kernel performance metrics at source level by SASS patching.
  • Provides support for automatically saving and restoring the functional state of the CUDA device.

For more info, go through the official documentation.
